# Day 2: Starting with AI-Generated Code — What Lovable Got Right, What It Got Wrong

**Series:** Building a Production SaaS in Public — 45 Days, 45 Posts
**Tags:** #buildinpublic #ai #lovable #react #codegen

---

I'll say the quiet part out loud: the first version of this entire platform was generated by AI.

Lovable is a tool where you describe what you want in natural language, and it generates a full React application — components, routes, state management, styling, the works. I described a coworking management platform with member portals, sales CRM, booking system, and admin dashboards. Twenty minutes later, I had a running app.

And I would absolutely do it again.

## What Lovable Generated

The output was a TanStack Start application — server-side rendered React with file-based routing. It included:

- A complete route tree with 30+ routes
- A Zustand store with seed data for branches, users, plans, leads, bookings, invoices — the full data model
- shadcn/ui components for every form, table, card, and dialog
- Tailwind CSS styling that was... actually pretty good
- Working navigation between all routes
- Forms that managed state and submitted data

The component hierarchy was solid. It understood that a coworking platform needs a public marketing site AND a private dashboard, and it separated them correctly. The seed data was realistic enough to demo — fake branch names, fake users with different roles, fake leads in various pipeline stages.

## What It Got Wrong

**Architecture decisions that didn't survive contact with reality:**

1. **TanStack Start with SSR.** The app was set up for server-side rendering. But 27 of my 33 routes are behind authentication. SSR adds a server, complicates deployment, and provides zero SEO benefit for dashboard pages. I ripped it out entirely in commit 6 and went pure SPA.

2. **A monolithic store.** Everything — types, seed data, business logic, UI state — lived in one 1,200-line file called `store.ts`. This is fine for a prototype. It's not fine when you need to debug why a lead's score isn't calculating correctly at 11pm.

3. **No backend.** The generated app stored everything in Zustand with localStorage persistence. Clear your browser, lose your data. Fine for a demo, not for a business.

4. **Overly complex components.** Some generated components had 6 levels of conditional rendering that could have been 3 separate components. AI tends to inline everything rather than extract.

5. **Missing accessibility.** No aria-labels. No keyboard navigation on custom components. No focus management on modals.

**The subtle bugs:**

- Form state that didn't reset after submission
- Inconsistent date formatting (some ISO, some locale-specific)
- TypeScript `any` types sprinkled in data transformation functions
- CSS that worked on desktop but broke on mobile
- Import paths that would break if you moved a file

## What I Kept

Here's the thing nobody tells you about AI code generation: the value isn't in the code. It's in the **decisions**.

Lovable made hundreds of micro-decisions that I would have spent days on:

- **Route structure.** Should the sales pipeline live at `/staff/sales` or `/dashboard/sales`? Should each lead have its own route (`/staff/sales/$leadId`) or open in a modal? The AI made reasonable choices, and I kept most of them.

- **Data model.** What fields does a Lead need? What's the relationship between a Membership, a Plan, a Branch, and a User? The generated types were 80% correct. I refined them, but the skeleton was right.

- **Component library choice.** shadcn/ui was the right call. I've been using it on other projects and the AI happened to pick it too. Lucky, but also validation.

- **UI patterns.** Kanban boards for the sales pipeline. Tables for invoices. Cards for branch listings. KPI cards for dashboard overviews. Badge components for statuses. These patterns are correct for this domain.

- **Seed data.** Realistic fake data — 6 branches with Indian addresses, users with Indian names, pricing in INR, leads with realistic company names and budget ranges. This saved me hours of writing `const mockBranch = { name: "Test Branch", ... }` over and over.

## The Rewrite Ratio

If I had to put a number on it: I kept about 30% of the generated code and rewrote 70%.

But that 30% was the expensive 30% — the part that requires domain knowledge, design decisions, and the kind of boring-but-necessary seed data work that nobody wants to do.

The 70% I rewrote was the structural stuff — state management architecture, API layer, authentication, layout system, component extraction. The stuff that requires software engineering judgment, not just code generation.

## The Mental Model That Works

Here's how I think about AI-generated code now:

**It's a first draft, not a final product.**

When a writer uses a first draft, they don't ship it. They use it to figure out the structure, identify the themes, and find the voice. Then they rewrite. Sometimes they keep whole paragraphs. Sometimes they keep one good sentence from three pages.

Lovable gave me a first draft of a coworking platform. The structure was there. The themes (public site, member portal, staff tools, admin panel) were correct. The voice (shadcn/ui aesthetic, card-based layouts, badge-heavy status displays) worked.

I kept the structure. I rewrote the internals. And the whole project shipped faster because of it.

## Would I Do It Again?

Without hesitation. Here's why:

1. **Speed to demo.** I had a working prototype to show the business team in 20 minutes. This bought me credibility and time to do the real engineering.

2. **Decision scaffolding.** Hundreds of small decisions were made for me. Even the wrong ones were useful — they showed me what wouldn't work, which is faster than starting from nothing.

3. **Seed data.** I cannot overstate how much time realistic seed data saves. Every screenshot, every demo, every test is more convincing with data that looks real.

4. **Confidence in scope.** Seeing the full app generated — all 33 routes, all 9 roles — confirmed that the scope was achievable. It's one thing to write a requirements doc. It's another to see it running.

The key insight: **AI code generation is a prototyping tool, not a shipping tool.** Use it to explore the problem space. Then engineer the solution yourself.
